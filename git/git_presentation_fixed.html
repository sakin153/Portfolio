Git. Введение

Система контроля версий для современной разработки

Что такое система контроля версий?

  • Система контроля версий — это система, регистрирующая изменения в одном или нескольких файлах с тем, чтобы в дальнейшем была возможность вернуться к определённым старым версиям этих файлов
  • Основные возможности:
    • Возврат отдельных файлов к состоянию на определённый момент времени
    • Возврат всего проекта к состоянию на определённый момент времени
    • Сравнение изменений с течением времени
    • Просмотр кто последний модифицировал что-то
    • Восстановление файлов

Зачем нужно знать системы контроля версий?

  • Обеспечивают совместную работу над проектом — несколько разработчиков могут вносить изменения параллельно
  • Позволяют хранить историю изменений, возвращаться к предыдущим версиям
  • Упрощают интеграцию кода через удалённый репозиторий
  • Являются основой процессов CI/CD: автоматические тесты, проверки и сборка на каждый коммит
  • Повышают качество и надёжность разработки

Локальные системы контроля версий

  • Простейший метод — копирование файлов в другую директорию (возможно, датированную)
  • Проблемы такого подхода:
    • Легко забыть в какой директории вы находитесь
    • Случайная запись не в тот файл
    • Случайное копирование не тех файлов
  • Локальные VCS: простая база данных, которая ведёт учёт всех изменений в файлах
  • Популярная система: RCS (Revision Control System) — хранит на диске наборы патчей (различий между файлами) в специальном формате

Централизованные системы контроля версий

  • Проблема: необходимость совместной работы с разработчиками на других системах
  • Решение: Централизованные системы контроля версий (CVS, Subversion, Perforce)
    • Единый сервер с файлами всех версий
    • Клиенты извлекают файлы из центрального хранилища
  • Преимущества:
    • Каждый знает что делают остальные участники проекта
    • Администраторы имеют полный контроль
    • Легче администрировать CVCS чем локальные базы
  • Недостатки:
    • Единая точка отказа
    • При недоступности сервера никто не может сохранять изменения
    • Повреждение центральной базы = потеря всей истории проекта

Распределённые системы контроля версий

  • Принцип: клиенты не просто извлекают снимок последних версий файлов — они полностью копируют весь репозиторий
  • Преимущества:
    • При отказе любого сервера любой клиентский репозиторий может быть скопирован обратно на сервер
    • Каждое извлечение данных — полное резервное копирование всех данных
    • Возможность работать с несколькими удалёнными репозиториями
  • Примеры DVCS: Git, Mercurial, Bazaar, Darcs
  • Модели совместной работы: можно настроить несколько типов рабочих процессов, невозможных в централизованных системах

Краткая история Git

  • 1991-2002: Ядро Linux разрабатывалось через передачу патчей и архивных файлов
  • 2002: Проект ядра Linux начал использовать проприетарную DVCS BitKeeper
  • 2005:
    • Отношения между сообществом разработчиков ядра Linux и компанией BitKeeper испортились
    • Статус бесплатного использования инструмента был отозван
  • Линус Торвальдс создал Git со следующими целями:
    • Скорость
    • Простая архитектура
    • Сильная поддержка нелинейной разработки (тысячи параллельных веток)
    • Полная распределённость
    • Способность эффективно управлять большими проектами (скорость и размер данных)

Основы Git

  • Снимки, а не различия:
    • Другие системы хранят информацию как список изменений файлов
    • Git хранит данные как снимки миниатюрной файловой системы
    • При каждом коммите Git сохраняет снимок всех файлов
  • Почти все операции локальны:
    • История проекта полностью доступна на локальном диске
    • Большинство операций мгновенные
    • Возможность работать офлайн
  • Целостность:
    • Контрольные суммы SHA-1
    • Невозможность незаметного повреждения файла
  • Git обычно только добавляет данные — сложно заставить систему удалить данные

Три состояния файлов в Git

Modified

Файл изменён, но ещё не зафиксирован

Staged

Изменённый файл отмечен для включения в следующий коммит

Committed

Данные безопасно сохранены в локальной базе

  • Рабочая директория — извлечённая из базы конкретная версия проекта
  • Область индексирования — файл с информацией о том, что войдёт в следующий коммит
  • Директория Git — место хранения метаданных и базы данных объектов

Установка Git

Linux

Debian/Ubuntu:
$ sudo apt install git-all

Fedora:
$ sudo dnf install git-all

macOS

Homebrew:
$ brew install git

Xcode:
$ git --version
(автоматическая установка)

Windows

Официальный сайт:
https://git-scm.com/download/win

Chocolatey:
choco install git

Первоначальная настройка Git

  • Настройка имени пользователя:
    $ git config --global user.name "John Doe"
  • Настройка email:
    $ git config --global user.email johndoe@example.com
  • Настройка текстового редактора:
    $ git config --global core.editor nano
  • Проверка настроек:
    $ git config --list
  • Получение справки:
    $ git help config
    $ git config --help
    $ man git-config

Получение справки

  • Три способа получить справку по любой команде Git:
    $ git help <команда>
    $ git <команда> --help
    $ man git-<команда>
  • Примеры:
    $ git help config
    $ git add --help
    $ man git-commit
  • Краткая справка (только опции):
    $ git add -h
